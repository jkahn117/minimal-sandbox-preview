# Slidev in Cloudflare Sandbox — Implementation Plan

## Goal

Replace the current Vue + Vite sandbox app with a [Slidev](https://sli.dev/)
presentation app. Instead of editing `App.vue` in the browser editor,
users will edit `slides.md` and see a live-updating slide deck in the
preview iframe.

---

## Background

### Current Setup

The sandbox runs a vanilla **Vite + Vue 3** app:

- `sandbox/app/` contains a standard Vite project (`index.html`,
  `src/main.ts`, `src/App.vue`, `vite.config.ts`)
- The Dockerfile (`docker.io/cloudflare/sandbox:0.7.4`) bakes in
  `node_modules` via `npm install` at build time
- At runtime, `src/lib/sandbox.ts` starts Vite via `npx vite --port 3001`
- The editable file is `/workspace/app/src/App.vue`

### How Slidev Differs

Slidev is **fundamentally different** from a standard Vite+Vue app:

| Aspect           | Current (Vite+Vue)                  | Slidev                                        |
|------------------|-------------------------------------|-----------------------------------------------|
| Entry point      | `index.html` → `src/main.ts`       | `slidev` CLI generates everything internally  |
| Dev server       | `npx vite --port 3001`             | `npx slidev --port 3001`                      |
| Editable file    | `src/App.vue`                       | `slides.md`                                   |
| Vite config      | User-authored, full control         | Slidev-managed, user config merged in         |
| Default port     | N/A (we set 3001)                   | 3030 (we override to 3001 via CLI flag)       |
| Dependencies     | `vue`, `vite`, `@vitejs/plugin-vue` | `@slidev/cli` (~70 deps), `@slidev/theme-default`, `vue` |
| Vite plugins     | Just `@vitejs/plugin-vue`           | Vue, UnoCSS, Markdown-it, Shiki, unplugin-icons, etc. |
| HTML generation  | Static `index.html`                 | Generated by Slidev at startup                |
| Vue bootstrap    | `createApp(App).mount("#app")`      | Handled internally by Slidev                  |

Key implication: we **delete** `index.html`, `src/main.ts`, and `src/App.vue`.
Slidev replaces all of them. The only user-authored content file is `slides.md`.

### What `pnpm create slidev` Installs

The `create-slidev` scaffolder produces a minimal project:

```
your-slidev/
├── .npmrc              # shamefully-hoist=true, auto-install-peers=true
├── package.json        # @slidev/cli, @slidev/theme-default, @slidev/theme-seriph, vue
├── slides.md           # Main slide deck (editable)
├── components/         # Custom Vue components (optional)
│   └── Counter.vue     # Demo counter component
├── netlify.toml        # Deploy config (not needed)
├── vercel.json         # Deploy config (not needed)
└── README.md
```

**`package.json` from the template:**
```json
{
  "name": "untitled",
  "type": "module",
  "private": true,
  "scripts": {
    "build": "slidev build",
    "dev": "slidev --open",
    "export": "slidev export"
  },
  "dependencies": {
    "@slidev/cli": "^52.14.1",
    "@slidev/theme-default": "latest",
    "@slidev/theme-seriph": "latest",
    "vue": "^3.5.29"
  }
}
```

The `.npmrc` with `shamefully-hoist=true` is required because Slidev's
plugin resolution expects a flat `node_modules` layout (pnpm's strict
hoisting breaks it). Since our Dockerfile uses `npm install` (which
hoists by default), this mainly matters for `auto-install-peers=true`.

## Version Updates

### Cloudflare Sandbox SDK

The SDK and Docker image have been updated significantly since our
project was created.

| Component                      | Current  | Latest     | Delta           |
|--------------------------------|----------|------------|-----------------|
| `@cloudflare/sandbox` (npm)    | `0.7.4`  | **`0.7.17`** | 13 patch releases |
| `cloudflare/sandbox` (Docker)  | `0.7.4`  | **`0.7.17`** | 13 patch releases |

### Key Fixes Between 0.7.4 and 0.7.17

Relevant to our use case:

- **0.7.6**: Improved `writeFile()` performance (native container writes
  instead of shell pipelines). Fixed `proxyToSandbox` redirect handling.
- **0.7.7**: Improved `readFile()` performance (native syscall reads).
  Fixed `proxyToSandbox` to pass redirect responses instead of following them.
- **0.7.8**: Fixed local dev crash loops after Docker restarts/idle
  timeouts (stale container state auto-recovery).
- **0.7.11**: Fixed heredoc commands hanging permanently. Fixed file watch
  race condition and SSE stream handler issues.
- **0.7.12**: **Fixed sandbox creation timing out for large container
  images** — transport retry budget now scales to configured startup
  timeouts. (Critical for Slidev's larger Docker image.)
- **0.7.13**: Fixed debounce deadlock in log pattern matching, incorrect
  HTTP 500 classification for transient startup errors, WebSocket chunk
  race on streaming responses.
- **0.7.14**: Fixed sandboxes staying awake after disabling `keepAlive`.
- **0.7.15**: Improved idle timeout handling for long-running streams.
  Added per-command and per-session timeouts for `exec`.
- **0.7.16**: Pass session env vars and working directory to PTY.
- **0.7.17**: Added `gitignore` and `excludes` options to `createBackup()`.

## Decisions

### 1. Include `vite.config.ts` with HMR/polling overrides

Slidev supports a user-level `vite.config.ts` that gets merged with its
internal config. We need two critical settings for sandbox proxy
compatibility:

- `server.hmr.clientPort: 443` — HMR WebSocket must connect through
  the Cloudflare proxy (browser sees port 443, container listens on 3001)
- `server.watch.usePolling: true` — Container filesystem doesn't
  support `inotify`; Vite must poll for file changes

**Risk:** Slidev docs warn this is "an advanced feature" and overrides
may "potentially cause the app to break." If Slidev's internal config
overrides our `server.hmr.clientPort`, HMR won't work. Fallback: use
env vars or investigate Slidev CLI flags.

### 2. Keep port 3001 (not Slidev's default 3030)

Override via `--port 3001` CLI flag. This avoids changes to:
- `wrangler.jsonc` container config
- `src/lib/sandbox-manager.ts` port references
- Preview URL patterns

### 3. Only `@slidev/theme-default` (not also `theme-seriph`)

Lighter Docker image. The official template includes both, but
`theme-default` is sufficient. Themes can be added later.

### 4. Upgrade `@cloudflare/sandbox` to 0.7.17

Both the npm package (in root `package.json`) and Docker base image
(in `sandbox/Dockerfile`). The 0.7.12 fix for large container image
timeouts is particularly important since Slidev's image will be
substantially larger.

## Implementation Steps

### Step 1 — Update root `package.json`

Bump `@cloudflare/sandbox` from `0.7.4` to `0.7.17`:

```json
"@cloudflare/sandbox": "0.7.17",
```

Run `pnpm install` to update `pnpm-lock.yaml`.

### Step 2 — Replace `sandbox/app/` contents

**Delete** these files (Slidev generates them internally):
- `sandbox/app/index.html`
- `sandbox/app/src/main.ts`
- `sandbox/app/src/App.vue`
- `sandbox/app/src/` (empty directory after deletion)

**Replace** `sandbox/app/package.json`:
```json
{
  "name": "sandbox-slidev",
  "private": true,
  "type": "module",
  "scripts": {
    "dev": "slidev --port 3001 --bind 0.0.0.0"
  },
  "dependencies": {
    "@slidev/cli": "^52.14.1",
    "@slidev/theme-default": "latest",
    "vue": "^3.5.29"
  }
}
```

**Create** `sandbox/app/.npmrc`:
```
shamefully-hoist=true
auto-install-peers=true
```

**Replace** `sandbox/app/vite.config.ts`:
```typescript
import { defineConfig } from 'vite'

export default defineConfig({
  server: {
    hmr: {
      clientPort: parseInt(process.env.VITE_HMR_CLIENT_PORT || "443"),
    },
    watch: {
      usePolling: true,
      interval: 500,
    },
  },
})
```

Note: Removed `cacheDir`, `plugins: [vue()]`, `host: "0.0.0.0"`, and
`allowedHosts: true` — Slidev manages these internally. The `host` binding
is handled by `--bind 0.0.0.0` CLI flag. We only override the settings
that Slidev doesn't know about (proxy HMR port and polling).

**Create** `sandbox/app/slides.md`:
```markdown
---
theme: default
---

# Welcome to Slidev

Presentation slides for developers

---

# Slide 2

- Edit this file in the editor
- Save to see live updates
- Add slides with `---` separator

---

# Code Example

​```ts
console.log('Hello, Slidev!')
​```

---

# Learn More

[Slidev Documentation](https://sli.dev)
```

### Step 3 — Update `sandbox/Dockerfile`

```dockerfile
FROM docker.io/cloudflare/sandbox:0.7.17

# Pre-bake the Slidev app so containers start instantly with no
# network dependency. Only slides.md is written at runtime (the editable file).

WORKDIR /workspace/app

# Copy package.json and .npmrc first for layer caching
COPY app/package.json app/.npmrc ./
RUN npm install

# Copy the rest of the app scaffold (slides.md, vite.config.ts)
COPY app/ ./

# Expose port 3001 for the Slidev dev server
# NOTE: Port 3000 is reserved by Sandbox internal Bun server
EXPOSE 3001

WORKDIR /workspace

# The base image already has ENTRYPOINT configured
# Don't override it - the sandbox binary will start automatically
```

Changes from current:
- Base image bumped from `0.7.4` to `0.7.17`
- Comments reference Slidev instead of Vue
- `.npmrc` is copied alongside `package.json`
- `npm install` now pulls `@slidev/cli` and its ~70 dependencies + theme

**Docker image size impact:** The current image installs `vue`, `vite`,
and `@vitejs/plugin-vue` (~50MB node_modules). Slidev will install
`@slidev/cli` with ~70 direct deps including Shiki, UnoCSS, unplugin-icons,
Monaco Editor, etc. Expect **~300-500MB node_modules** inside the image.
Build time will increase but runtime startup is unaffected (everything
is baked in).

### Step 4 — Update `src/lib/sandbox.ts`

Key changes:

1. `EDITABLE_FILE` → `/workspace/app/slides.md`
2. `DEFAULT_APP_VUE` → `DEFAULT_SLIDES_MD` (default slide deck content)
3. Start command: `npx vite --port 3001` → `npx slidev --port 3001 --bind 0.0.0.0`
4. Token: `"vuehmr"` → `"slidev"` (cosmetic, changes preview URL subdomain)
5. `portName`: `"vite-dev"` → `"slidev-dev"`
6. Health check: may need increased timeout (see Risk Areas below)

```typescript
const APP_DIR = "/workspace/app";
export const EDITABLE_FILE = `${APP_DIR}/slides.md`;

const DEFAULT_SLIDES_MD = `---
theme: default
---

# Welcome to Slidev

Presentation slides for developers

---

# Slide 2

- Edit this file in the editor
- Save to see live updates
- Add slides with \`---\` separator
`;

export const sandboxManager = new SandboxManager({
  port: 3001,
  token: "slidev",
  portName: "slidev-dev",
  sleepAfter: "5m",

  async initialize({ sandbox, progress }) {
    progress("writing_files");
    await sandbox.writeFile(EDITABLE_FILE, DEFAULT_SLIDES_MD);

    progress("starting_server");
    await sandbox.startProcess("npx slidev --port 3001 --bind 0.0.0.0", {
      processId: "slidev-dev",
      cwd: APP_DIR,
      env: {
        PORT: "3001",
        NODE_ENV: "development",
        CHOKIDAR_USEPOLLING: "true",
        CHOKIDAR_INTERVAL: "500",
      },
    });

    await new Promise((resolve) => setTimeout(resolve, 2000));

    // ... same process crash check and health check loop as current ...
  },
});
```

## Risk Areas

### 1. HMR Through the Proxy

**Risk:** Slidev manages its own Vite config internally and merges the
user-level `vite.config.ts`. If Slidev's internal config overrides our
`server.hmr.clientPort: 443`, the HMR WebSocket in the iframe will try
to connect to port 3001 (the internal container port) instead of 443
(the proxy port), and HMR will fail silently.

**Mitigation:** Vite's config merging strategy for `server.hmr` is
object-level merge, not full replacement — our `clientPort` should
survive. If not, we can try the `VITE_HMR_CLIENT_PORT` env var approach
or investigate whether Slidev exposes an HMR port configuration.

**Verification:** After deployment, open browser DevTools → Network →
WS tab. Verify the HMR WebSocket connects to `wss://<preview-host>:443/`
not `ws://localhost:3001/`.

### 2. Startup Time

**Risk:** Slidev's first cold start is significantly heavier than vanilla
Vite. It resolves themes, compiles Shiki grammars, sets up UnoCSS,
generates virtual modules, etc. The current health check loop is
15 attempts × 3 seconds = **45 seconds max**.

**Mitigation:** May need to increase to 20 attempts × 3 seconds (60s)
or 25 × 3s (75s). The 0.7.12 sandbox SDK fix ensures the transport
retry budget scales to our timeout, so the container won't time out
before our health check loop completes.

**Verification:** Monitor the health check loop in logs. If it reaches
15 attempts and fails, increase the limit.

### 3. Docker Image Size & Build Time

**Risk:** `@slidev/cli` has ~70 direct dependencies. `npm install` during
Docker build will take significantly longer (2-5 minutes vs ~30 seconds).
The Docker image will be ~300-500MB larger.

**Mitigation:** This only affects build time, not runtime. The image is
built once and cached. Layer caching (`package.json` copied before
`npm install`) minimizes rebuilds.

### 4. File Watching in the Container

**Risk:** Slidev uses Vite's file watching under the hood. Our
`vite.config.ts` sets `usePolling: true`, but if Slidev's internal config
creates the watcher before merging our config, the initial watcher won't
poll.

**Mitigation:** The `CHOKIDAR_USEPOLLING` and `CHOKIDAR_INTERVAL` env
vars in the start command provide a belt-and-suspenders fallback — they
configure chokidar globally regardless of Vite config merge order.

**Verification:** Edit `slides.md` in the editor, save, and verify the
preview updates within 1-2 seconds.

### 5. Slidev UI Elements

**Risk:** Slidev's dev server includes navigation controls, presenter
mode buttons, and a sidebar that may not work correctly when embedded in
an iframe behind a proxy. These are non-essential but could cause
confusion.

**Mitigation:** Acceptable for a prototype. Slidev's presenter mode and
remote features won't work through the sandbox proxy, but the slide
content and navigation will render correctly. If needed, custom CSS can
hide the Slidev toolbar.
